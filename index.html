Problem

Analysts working with raw CSV exports had no local, repeatable tool for KPI extraction and anomaly detection. Every analysis required manual scripting, cloud tooling, or ad hoc Jupyter notebooks — none of which produced consistent, structured output.

System

Raw file ingestion → Schema detection → KPI extraction → Anomaly scoring → Group comparison → Report generation

  • Accepts CSV, Excel, and JSON. Schema detection maps columns to known metric types without manual configuration.
  • KPI selection layer identifies primary and derived metrics based on field types and statistical distributions.
  • Anomaly detection flags outlier concentrations and segment imbalances with driver attribution.
  • Group comparison engine cross-segments by categorical fields (VendorID, Borough, flag status) to isolate signal from noise.
  • Report writer produces structured HTML dashboard and JSON findings file. Both land in outputs/latest/.
  • Full pipeline packaged as a standalone macOS application via PyInstaller — no Python install, no network required.
Output
  • 2M+ rows processed locally with no cloud dependency
  • 20 findings generated from TLC dataset — quality score 70/100
  • 496,218 outlier rows flagged with source attribution
  • Structured dashboard HTML and machine-readable JSON findings in outputs/latest/

03 Systems · AI Workflows

AI Ops
Engine

Problem

AI-assisted development across multiple project contexts produced inconsistent agent behavior, lost session state between conversations, and no durable record of decisions or system architecture. Each session started from scratch.

System

Hook dispatch → Memory retrieval → Task execution → Decision logging → Routine verification

  • 3-layer OS model: control plane at repo root, OS implementation in /OS/, engine tenants isolated under /engines/. Each layer has defined responsibilities and cannot absorb concerns from the others.
  • Hook dispatch: pre/post-tool hooks intercept agent actions and update system state. Git hooks enforce documentation at commit and push boundaries.
  • Memory system with three tiers: raw observations, processed daily summaries, and promoted durable knowledge articles with indexed domain maps.
  • 5 automated daily routines (R01–R05) covering job-search gate checks, doc sync, drift detection, system status, and validation. Run at 08:00–08:40 CDT via scheduled Claude sessions.
  • Rule shards in /.claude/rules/ load automatically at session start — enforcing retrieval order, memory protocol, and ask outbox discipline without requiring manual instruction.
Output
  • Production governance layer running daily — not a prototype
  • Session context persisted across conversations through indexed memory articles
  • 5 daily automated routines producing structured status outputs to out/asks/
  • Decision log, checkpoint log, and active plan maintained as live documents
  • Engine tenant (Analytics Runtime) fully isolated with its own pipeline, outputs, and CLAUDE.md

Mechanic
Shop POS

Problem

Mechanic shops operating with paper-based or spreadsheet work orders had no unified system linking job intake, parts tracking, invoicing, and customer history across the shop lifecycle.

System

Work order intake → Parts tracking → Status management → Invoice generation → Customer history

05 Mobile · React Native

MyGrocer

Problem

Shared grocery lists across household members required manual refresh to stay current — changes made on one device weren't visible on another until the app was reopened.

System

List edit → Firebase write → Realtime propagation → Device sync → Suggestion generation

  • Firebase Realtime Database with listener-based sync — changes write and propagate in under 200ms across all connected devices.
  • Purchase history model feeds a suggestion engine that surfaces frequently bought items and category-based recommendations at list creation.
  • Categorized list view (Produce, Dairy, Bakery, Meat) with per-category collapse, item quantity tracking, and store-aisle ordering.
  • React Native / Expo — cross-platform iOS and Android from a single codebase. v1.0 shipped.
Output
  • Under 200ms sync latency across multiple devices
  • Smart suggestions drawn from purchase history at list creation time
  • v1.0 shipped — iOS and Android via Expo
